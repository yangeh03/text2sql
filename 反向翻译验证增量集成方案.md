# 反向翻译验证（Back-Translation Validation）增量集成方案

参考资料：`re-act/2509.12612v1.pdf`

本方案在现有 Text-to-SQL LangGraph 智能体（preprocess → generate → validate → revise）中，增量加入“反向翻译验证”步骤：在模型生成 SQL 后，用 LLM 将 SQL 翻译回自然语言，与原问题进行语义一致性比对；一致则通过，不一致则触发修订。这一步可作为“执行验证前的语义闸门”，减少无效执行与无意义修订。

---

## 1. 需求梳理（用于 10 分钟课堂演示）

- **场景与痛点**：
  - 现有系统通过“执行引导 + 错误修订”显著提升可执行率，但仍存在“语义偏差”问题：SQL 能执行但不回答原问题。
  - 执行验证成本较高（连接数据库 + 真实执行），且错误定位较粗粒度。
- **核心想法**（论文思路）：
  - 生成 SQL 后，先让 LLM 将 SQL 反向翻译为自然语言描述（意图、约束、聚合等），再与原问题进行语义一致性判断。
  - 若一致，继续执行或直接接受；若不一致，向 LLM 提供差异点进行修订（优先修语义，再修执行）。
- **目标**：
  - 增量集成一个“语义一致性闸门”，不大改现有流程与模块。
  - 支持可配置：只做语义闸门（不替代执行验证）或在 Demo 模式下可短路直接输出（课堂演示友好）。
- **期望收益**：
  - 降低“语义错而执行对”的无意义成功，提高最终答案正确性。
  - 通过语义差异的可解释反馈，提升修订的命中率。

---

## 2. 实现思路（尽量小改，模块增量）

- **新增一个 LangGraph 节点：`semantic_validate`**
  - 位置：`generate_sql` 之后、`validate_sql` 之前。
  - 功能：
    1) `back-translate`：将 SQL → NL 语义描述（JSON结构化输出）。
    2) `judge`：对比原问题与反向描述，打出语义相似分（0~1），返回是否通过。
  - 结果字段写回 `AgentState`：`back_translation`、`semantic_score`、`semantic_pass`、`semantic_reason`。
- **路由策略（可配置）**
  - 默认：`generate_sql → semantic_validate → validate_sql`（仅作为前置闸门，不改变原有执行验证）
  - Demo/快速模式（可选）：若 `semantic_pass=True` 且 `SEMANTIC_GATE_MODE=finalize_on_pass`，则可直接 `finalize` 快速出结果（课堂演示时可用）。
  - 失败：`semantic_pass=False` → 进入 `revise_sql`，同时把语义差异（reason/对齐建议）喂给 LLM。
- **最小改动点**
  - 新增 prompts（2段）：`BACK_TRANSLATE_PROMPT`、`SEMANTIC_JUDGE_PROMPT`。
  - 新增节点函数：`semantic_validate_node(state)`。
  - 轻量调整 `AgentState`：新增 3~4 个字段。
  - 在 `build_graph()` 中插入一个节点与条件边；`decide_next_step` 维持原有逻辑，新增 `decide_after_semantic`（仅用于从 `semantic_validate` 路由到 `revise_sql` 或 `validate_sql/finalize`）。
  - `config.Settings` 新增开关与阈值（环境变量覆盖）：
    - `ENABLE_SEMANTIC_VALIDATE`（默认 true/false 皆可）
    - `SEMANTIC_SCORE_THRESHOLD`（默认 0.7）
    - `SEMANTIC_GATE_MODE` in {`before_exec`(默认), `finalize_on_pass`}

## 3. 伪代码 + 可视化与指标示例

### 3.1 节点伪代码（semantic_validate）

```python
# 新增到 graph.py

def semantic_validate_node(state: AgentState) -> AgentState:
    sql = state.sql_draft
    schema = state.relevant_schema or state.db_schema

    # 1) Back-translate SQL → NL(JSON)
    back_prompt = BACK_TRANSLATE_PROMPT.format(sql=sql, schema=schema)
    bt = chat(messages=[{"role": "user", "content": back_prompt}], temperature=0.1)
    bt_json = safe_parse_json(bt.get("content", ""))  # { intent, tables, columns, filters, agg, group_by, order_by, limit }

    # 2) Judge 语义一致性
    judge_prompt = SEMANTIC_JUDGE_PROMPT.format(question=state.question, bt=json.dumps(bt_json, ensure_ascii=False))
    jd = chat(messages=[{"role": "user", "content": judge_prompt}], temperature=0.0)
    jd_json = safe_parse_json(jd.get("content", ""))  # { pass: bool, score: float, reason: str }

    state.back_translation = bt_json
    state.semantic_score = float(jd_json.get("score", 0))
    state.semantic_pass = bool(jd_json.get("pass", False))
    state.semantic_reason = jd_json.get("reason", "")

    # 记录轨迹
    state.step_trace.append(f"semantic: score={state.semantic_score} pass={state.semantic_pass}")
    return state
```

### 3.3 流程描述（文本）

- preprocess（预处理）
  - 功能：汇集生成所需上下文。
  - 动作：
    - 获取完整数据库模式（CREATE TABLE 语句）。
    - 模式链接：从完整模式中筛选与问题相关的表/列，形成精简 schema。
    - 检索示例值：结合 LLM 关键值抽取和数据库查询，收集实体值样例。
  - 产出：`db_schema`、`relevant_schema`、`retrieved_values`。

- generate_sql（生成 SQL）
  - 功能：基于问题、相关 schema、示例值生成 SQL 草稿。
  - 产出：`sql_draft`（单行 SQL）。

- semantic_validate（反向翻译语义验证）
  - 功能：用 LLM 将 `sql_draft` 反向翻译为自然语言结构化描述，并与原问题做语义一致性判定。
  - 动作：
    - Back-translate：SQL → JSON（如 intent/tables/columns/filters/agg/...）。
    - Judge：输出 `{pass, score, reason}`，标注是否语义一致与不一致原因。
  - 产出：`back_translation`、`semantic_pass`、`semantic_score`、`semantic_reason`。

- gate mode（语义闸门路由）
  - 若 `semantic_pass=False`（或 `score<threshold`）：进入 `revise_sql` 按语义差异修正。
  - 若 `semantic_pass=True`：
    - `before_exec`（默认）：进入 `validate_sql` 执行验证。
    - `finalize_on_pass`：直接 `finalize`（课堂演示/快速出结果）。

- revise_sql（修订 SQL）
  - 功能：依据执行错误和语义差异提示修正 SQL。
  - 动作：在提示中注入错误信息与 `[语义差异提示]`，指导 LLM 对齐问题意图、表/列/过滤/聚合等。
  - 产出：更新后的 `sql_draft`，写入 `revision_history`。

- validate_sql（执行验证）
  - 功能：在真实数据库执行 SQL，校验是否可执行以及是否返回结果。
  - 产出：`execution_result={status, rows, rowcount | error}`，并记录入 `revision_history`。

- decide_next_step（执行后的路由）
  - 若 `status=error`：进入 `revise_sql`。
  - 若 `status=ok` 且 `rowcount=0`：允许一次针对“空结果”的修订，再次不变则接受空结果。
  - 若 `status=ok` 且 `rowcount>0`：进入 `finalize`。

- finalize（最终化）
  - 功能：确认最终 SQL。
  - 动作：设置 `final_sql=sql_draft`，结束流程。

- report_metrics（自动汇总指标）
  - 功能：在一轮评测完成后，自动读取 `outputs/semantic_reports/*.json` 与运行日志，聚合生成可直接展示的指标与明细表。
  - 产出：
    - `outputs/reports/summary.json`：总体聚合指标（SPR/Exec/SCE/Avg Revisions/Empty-result/Disagreement 等）。
    - `outputs/reports/details.csv`：逐样本明细（便于做表格/筛选/画图）。
    - `outputs/reports/report.md`：用于课堂/汇报的 Markdown 表格与要点摘要。

### 3.4 指标与表格示例（展示用）

- 语义闸门通过率（Semantic Pass Rate, SPR）= 通过语义验收的样本 / 总样本
- 语义-执行一致成功率（Semantically-correct & Executable, SCE）= 通过语义 + 执行成功的样本 / 总样本
- 修订效率提升（Avg Revisions ↓）：引入语义闸门后平均修订次数的下降幅度

| 指标 | 引入前 | 引入后 | 变化 |
|---|---:|---:|---:|
| SPR | - | 78.2% | +78.2% |
| 执行成功率 | 62.5% | 68.9% | +6.4% |
| Avg Revisions | 1.8 | 1.2 | -0.6 |

（注：以上数值为示例展示，课堂演示时可替换为实际跑分结果）

## 8. 演示（PPT）建议大纲（10 分钟）

1) 背景与问题（1.5min）
- 执行验证的局限：语义错也可能执行对
- 目标：最小代价提升语义准确性

2) 核心方法（2min）
- 反向翻译 + 语义判定（结构化）
- 路由策略与配置（before_exec / finalize_on_pass）

3) 体系集成（2min）
- LangGraph 新增 `semantic_validate` 节点
- 边与条件路由说明（Mermaid 图）

4) 实验与指标（2.5min）
- 展示 3~5 个真实样例（问题/SQL/反译/判定/修订）
- 表格：SPR / 执行成功率 / Avg Revisions（示例数值 + 解释）

具体查看位置与方法（新增“自动聚合”产物）：

- 产物路径
  - 预测 SQL：`outputs/predictions.sql`
  - 终端日志：`outputs/logs/semantic_smoke.log`（由冒烟脚本写入）
  - 语义报告（开启语义闸门后）：`outputs/semantic_reports/*.json`
    - 每条样本一个 JSON，含 `question/sql/back_translation/score/pass/reason`
  - 指标聚合产物（自动汇总）：`outputs/reports/`
    - `summary.json`（总体指标）
    - `details.csv`（逐样本明细）
    - `report.md`（可直接贴到 PPT 的指标表格）

- 快速查看
  - 查看前 N 条 SQL：
    ```bash
    head -n 10 outputs/predictions.sql
    ```
  - 查看语义报告样例（若开启 ENABLE_SEMANTIC_VALIDATE）：
    ```bash
    ls -1 outputs/semantic_reports | head -n 5
    sed -n '1,80p' outputs/semantic_reports/0001_*.json
    ```
  - 查看运行日志：
    ```bash
    sed -n '1,120p' outputs/logs/semantic_smoke.log
    ```

- 指标定义（展示时逐条解释；summary.json 将直接给出这些值）
  - SPR（Semantic Pass Rate，语义通过率）
    - 定义：语义判定 `pass=true` 的样本数 / 总样本数。
    - 体现：反向翻译后，与原问题语义一致的比例。
  - Exec Success（执行成功率）
    - 定义：数据库执行 `status=ok` 的样本数 / 总样本数（不含语义判定）。
    - 体现：可执行性与语法/表列选择的正确性。
  - SCE（Semantically-correct & Executable）
    - 定义：`pass=true` 且执行 `status=ok` 的样本数 / 总样本。
    - 体现：同时满足语义一致与可执行的综合质量。
  - Avg Revisions（平均修订次数）
    - 定义：每条样本的修订次数均值（`revision_history` 的长度均值）。
    - 体现：语义闸门是否减少无效修订、提升一次成功概率。
  - Validity Rate（可执行率）
    - 定义：执行不报错的样本数 / 总样本数；与 Exec Success 类似，可按需要合并展示。
  - Empty-result Rate（空结果率）
    - 定义：`status=ok` 且 `rowcount=0` 的样本占比；可体现 WHERE/JOIN 过严等问题。
  - Semantic Disagreement（语义不一致率）
    - 定义：`pass=false` 的样本占比；可结合 `reason` 给出典型错误类别统计。

- 课堂快速统计方法（示例）
  - 基于语义报告（已开启语义闸门）：
    ```bash
    # 语义通过率 SPR（需要 jq；如未安装，可用 Python 方案）
    jq -s '[.[].semantic.pass] | map(select(.==true)) | length as $p | length as $t | {SPR: ($p/$t)}' outputs/semantic_reports/*.json
    ```
    ```bash
    # Python 版（无 jq 环境时）
    python - <<'PY'
import json,glob
files=glob.glob('outputs/semantic_reports/*.json')
if not files:
    print('No semantic reports found. Enable ENABLE_SEMANTIC_VALIDATE=true first.')
    raise SystemExit
data=[json.load(open(f,'r',encoding='utf-8')) for f in files]
spr=sum(1 for x in data if x.get('semantic',{}).get('pass'))/len(data)
print({'SPR': round(spr,4)})
PY
    ```
  - 平均修订次数（无需开启语义闸门，直接用控制台日志或在 main 中扩展统计）：
    ```bash
    # 简易法：从日志中抓取“📝 修订次数: k”，快速估算
    grep -o '修订次数: [0-9]\+' outputs/logs/semantic_smoke.log | awk '{sum+=$2; n+=1} END{if(n) printf("Avg Revisions: %.2f\n", sum/n); else print("No revision records")}'
    ```

- 展示建议
  - 选择 3~5 条报告 JSON，展示 `question/sql/back_translation/score/pass/reason`，直观说明语义闸门的价值。
  - 表格列建议：`样本ID | DB | pass | score | exec(ok/err) | rowcount | revisions | reason(截断)`。
  - 若时间允许，给出“引入前/后”的对比：SPR、Exec Success、Avg Revisions 的变化（可用 20~50 条样本做小规模实验）。

## 新增：自动聚合指标与报表（report_metrics）

- 触发时机：每次 `python src/agent/main.py` 跑完一轮评测后自动执行（代码实现将在后续提交）。
- 输入数据：
  - 语义报告：`outputs/semantic_reports/*.json`
  - 控制台日志：`outputs/logs/semantic_smoke.log`
  - 预测 SQL：`outputs/predictions.sql`
- 输出产物（用于直接展示/对比）：
  - `outputs/reports/summary.json`
    - 字段：
      - `total` 总样本数
      - `spr` 语义通过率（0~1）
      - `exec_success` 执行成功率（0~1）
      - `sce` 语义通过且执行成功的占比（0~1）
      - `avg_revisions` 平均修订次数
      - `empty_result_rate` 空结果占比
      - `semantic_disagreement` 语义不一致占比
  - `outputs/reports/details.csv`
    - 列：`index,db_id,question,sql,semantic_pass,semantic_score,semantic_reason,exec_status,rowcount,revisions`
  - `outputs/reports/report.md`
    - 包含一个可复制到 PPT 的表格示例与关键结论摘录。

- 示例（report.md 中的表格片段）：

| 样本ID | DB | pass | score | exec | rows | revisions | reason |
|---:|---|:---:|---:|:---:|---:|---:|---|
| 1 | soccer_3 | ✅ | 1.00 | ✅ | 1 | 1 | - |
| 2 | soccer_3 | ✅ | 1.00 | ✅ | 1 | 1 | - |
| 3 | soccer_3 | ✅ | 1.00 | ✅ | 9 | 1 | - |

- 调用方式（后续代码实现后将自动生成；也可单独运行聚合脚本）：
  ```bash
  # 方式A：随 main.py 自动执行（推荐课堂演示）
  python src/agent/main.py

  # 方式B：手动运行聚合（可重复执行以更新报表）
  python src/agent/report_metrics.py \
    --reports-dir outputs/semantic_reports \
    --log-file outputs/logs/semantic_smoke.log \
    --pred-file outputs/predictions.sql \
    --out-dir outputs/reports
  ```

- 与现有流程的关系：
  - report_metrics 不改变 agent 推理与验证，只是对产物进行结构化汇总，保证“生成后即可一键出表”。

5) 工程实现与开关（1min）
- prompts/graph/config 的最小改动点
- 可扩展：结构化对齐字段、RAG 约束词典

6) 结论与展望（1min）
- 语义闸门提升质量与可解释性
- 与执行验证互补，不冲突
